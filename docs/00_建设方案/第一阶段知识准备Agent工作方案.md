# 第一阶段知识准备Agent工作方案

> 版本：v0.1  
> 适用阶段：IT治理知识库第一阶段知识准备  
> 适用范围：以操作风险管理矩阵、配套流程图、制度文件、OA流程、表单、系统清单、岗位职责等基础资料为输入，生成可人工审核、可入库、可追溯的知识资产。  
> Agent配置：先采用3个Agent，分别为“知识抽取Agent”“知识生成Agent”“知识质检Agent”。

---

## 1. 第一阶段目标

第一阶段不以直接问答为目标，而以完成知识库的基础知识准备为目标。

核心目标包括：

1. 将原始文件转换为可处理的结构化中间结果；
2. 从资料中抽取治理域、流程、流程节点、风险、控制、制度、岗位、系统、表单、证据等治理对象；
3. 建立对象之间的基础关系；
4. 生成可入库的 Markdown、YAML、Mermaid 文件；
5. 通过人工审核节点确认关键内容；
6. 形成第一版可导入语析的 IT 治理知识资产。

---

## 2. 总体工作流

```text
基础资料准备
  ↓
知识抽取Agent
  ↓
【人工审核节点A：抽取结果审核】
  ↓
知识生成Agent
  ↓
【人工审核节点B：生成文件审核】
  ↓
知识质检Agent
  ↓
【人工审核节点C：入库前确认】
  ↓
入库语析
  ↓
问答与检索测试
```

第一阶段建议先选择一个治理域试点，例如“系统需求管理”，跑通后再扩展到其他治理域。

---

## 3. 输入资料范围

### 3.1 第一批输入资料

| 资料类型 | 文件示例 | 用途 | 优先级 |
|---|---|---|---|
| 风险控制矩阵 | 操作风险管理矩阵_信息服务管理 | 抽取流程、风险、控制、证据、责任岗位、制度依据 | P0 |
| 流程图 | 流程图_信息服务管理 | 抽取泳道、节点、顺序、判断条件、系统标识 | P0 |
| 制度文件 | 项目管理办法、需求管理细则、授权细则 | 补充制度依据、审批规则、留痕要求 | P1 |
| OA流程 | OA流程清单、表单字段、审批节点 | 补充实际执行节点和表单要求 | P1 |
| 表单模板 | 需求申请、项目验收、需求结算等 | 补充输入输出和控制证据 | P1 |
| 系统清单 | OA、需求管理系统、问题管理系统等 | 补充支撑系统关系 | P2 |
| 岗位职责 | 信息服务部岗位职责 | 补充责任岗位和执行岗位 | P2 |

### 3.2 试点建议

第一轮建议只处理一个治理域：

```text
系统需求管理
```

试点原因：

1. 需求管理与IT治理日常工作强相关；
2. 流程边界相对清晰；
3. 风险控制矩阵和流程图均有对应内容；
4. 后续便于用真实问题测试语析问答效果。

---

## 4. 三个Agent分工

```text
知识抽取Agent
负责：从原始资料中抽取治理对象和初步关系。

知识生成Agent
负责：将审核后的抽取结果生成标准化知识文件。

知识质检Agent
负责：检查知识文件是否满足入库要求，并形成待确认清单。
```

三个Agent之间必须通过标准交付物衔接，不允许后一个Agent直接跳过前一个Agent的结果自行发挥。

# 5. 知识抽取Agent

## 5.1 Agent定位

知识抽取Agent是第一阶段的入口Agent，负责把原始资料转换为结构化抽取结果。

它不负责生成最终知识库文件，也不负责判断制度是否合理，只负责“从资料中抽取已有事实”。

## 5.2 输入

```text
1. 操作风险管理矩阵
2. 配套流程图
3. 制度文件
4. OA流程和表单
5. 系统清单
6. 岗位职责
7. 本次指定的治理域或流程范围
```

## 5.3 输出

输出一个《知识抽取结果表》，建议为 Markdown 表格或 YAML/JSON。

必须包含：

```text
来源文件
来源位置
治理域
对象类型
对象名称
对象原文
抽取结果
初步对象编号
初步关系
置信度
待确认事项
```

## 5.4 工作流程

```text
步骤1：识别资料类型
  - 判断输入文件属于矩阵、流程图、制度、OA、表单、系统清单还是岗位职责。

步骤2：确定抽取范围
  - 只抽取本次指定治理域或指定流程范围。
  - 不做全量扩展。

步骤3：从风险控制矩阵抽取对象
  - 抽取一级流程、二级流程、三级流程。
  - 抽取风险点。
  - 抽取控制点。
  - 抽取主责部门、执行岗位、配合部门。
  - 抽取控制频率、控制方式、控制证据。
  - 抽取内部制度和外部监管制度。

步骤4：从流程图抽取对象
  - 抽取泳道角色。
  - 抽取流程阶段。
  - 抽取流程节点。
  - 抽取节点顺序。
  - 抽取判断条件。
  - 抽取系统标识。
  - 抽取上下游流程引用。

步骤5：从制度文件抽取对象
  - 抽取制度名称。
  - 抽取条款编号。
  - 抽取条款内容。
  - 抽取审批要求、责任要求、留痕要求、例外情形。

步骤6：从OA和表单抽取对象
  - 抽取流程名称、发起人、审批节点、审批角色、表单字段、附件要求、输出结果。

步骤7：生成初步对象编号
  - 治理域：DOM-XX
  - 流程：PROC-XX-XX
  - 节点：NODE-XX-XX-XX
  - 风险：RISK-XX-XX-XX
  - 控制：CTRL-XX-XX-XX
  - 制度：POL-XX
  - 表单：FORM-XX
  - 系统：SYS-XX
  - 岗位：ROLE-XX
  - 证据：EVD-XX-XX-XX

步骤8：标记置信度
  - 高：来源明确、字段完整、可直接追溯。
  - 中：来源明确但字段不完整。
  - 低：来源模糊、存在歧义或需人工判断。

步骤9：形成待确认事项
  - 标记冲突、缺失、重复、不清晰、不一致内容。
```

## 5.5 边界规则

知识抽取Agent必须遵守以下规则：

```text
1. 只能基于来源文件抽取，不得自行编造。
2. 不得自行新增流程、风险、控制、制度依据。
3. 不得自行删除重复对象，只能标记“疑似重复”。
4. 不得自行合并口径不同的流程节点。
5. 不得改变原始风险描述和控制点表述。
6. 不得自行调整主责部门、执行岗位、审批层级。
7. 不得判断制度条款是否有效，只能记录来源。
8. 不确定内容必须标记“待确认”。
9. 发现矩阵与流程图不一致时，输出冲突清单，不直接修正。
10. 每条抽取结果必须保留 source_file 和 source_location。
```

## 5.6 人工审核节点A：抽取结果审核

### 审核位置

知识抽取Agent完成《知识抽取结果表》后，必须暂停，进入人工审核。

### 审核人建议

```text
规划管理室负责人
对应流程owner
风险合规对接人
必要时邀请流程执行岗位人员
```

### 审核内容

| 审核项 | 审核问题 | 处理方式 |
|---|---|---|
| 范围 | 是否只抽取了本次指定治理域 | 超范围内容暂不进入下一步 |
| 对象 | 流程、风险、控制、证据是否抽全 | 补充或退回重抽 |
| 原文 | 是否保留原始表述 | 缺失原文的不进入下一步 |
| 来源 | 是否有来源文件和来源位置 | 无来源的不进入下一步 |
| 关系 | 初步关系是否明显错误 | 标记修正 |
| 冲突 | 矩阵和流程图是否冲突 | 列入冲突清单 |
| 待确认 | 是否存在需要业务确认的问题 | 明确责任人和确认结论 |

### 审核结论

人工审核后必须给出以下三类结论之一：

```text
通过：进入知识生成Agent。
有条件通过：修正指定问题后进入知识生成Agent。
退回：重新抽取。
```

# 6. 知识生成Agent

## 6.1 Agent定位

知识生成Agent负责把人工审核通过的抽取结果，转换为标准知识库文件。

它不直接读取未审核的原始资料，不自行改变审核后的对象和关系。

## 6.2 输入

```text
1. 人工审核后的知识抽取结果表
2. 人工审核结论
3. 对象编号规则
4. 流程文档模板
5. YAML对象模板
6. Mermaid图生成规则
```

## 6.3 输出

知识生成Agent必须输出三类文件：

```text
1. Markdown流程文档
2. YAML对象文件
3. Mermaid流程图文件
```

必要时输出：

```text
4. 问答知识片段
5. 变更日志
6. 待确认事项清单
```

## 6.4 工作流程

```text
步骤1：读取审核通过的抽取结果
  - 只处理审核结论为“通过”或“有条件通过且已修正”的内容。
  - 不处理审核结论为“退回”的内容。

步骤2：生成流程Markdown文档
  - 按流程维度生成文档。
  - 每个流程一个Markdown文件。
  - 包含基本信息、流程说明、流程节点、风险与控制、制度依据、控制证据、Mermaid流程图、常见问答、变更记录。

步骤3：生成YAML对象文件
  - 每个治理对象一个YAML。
  - 流程、节点、风险、控制、制度、系统、岗位、表单、证据分别存放。
  - 每个对象必须保留source_file、source_location、version、status。

步骤4：生成Mermaid流程图
  - 依据审核后的节点顺序生成。
  - 判断节点必须保留判断条件。
  - 复杂流程可拆成子流程。
  - 不确定节点必须用注释或待确认标记，不得强行串联。

步骤5：生成问答知识片段
  - 从流程、风险、控制、证据中生成基础问答。
  - 问答必须基于已审核内容。

步骤6：生成文件清单
  - 输出本次生成了哪些文件。
  - 标注建议入库路径。

步骤7：生成待确认事项
  - 对无法生成、信息缺失、关系不完整的内容列出待确认。
```

## 6.5 文件路径规则

建议路径如下：

```text
docs/03_流程文档/{治理域}/PROC-XX-XX_流程名称.md
docs/04_流程图_Mermaid/{治理域}/PROC-XX-XX_流程名称.md
docs/07_问答知识/{治理域}/QA-XX-XX_流程名称.md
objects/processes/PROC-XX-XX.yaml
objects/nodes/NODE-XX-XX-XX.yaml
objects/risks/RISK-XX-XX-XX.yaml
objects/controls/CTRL-XX-XX-XX.yaml
objects/policies/POL-XX.yaml
objects/systems/SYS-XX.yaml
objects/roles/ROLE-XX.yaml
objects/evidence/EVD-XX-XX-XX.yaml
changelog/CHANGELOG.md
```

## 6.6 边界规则

知识生成Agent必须遵守：

```text
1. 只能基于人工审核后的抽取结果生成文件。
2. 不得自行新增对象、删除对象、合并对象。
3. 不得改写风险和控制原始表述。
4. 不得自行补全制度依据、审批层级、责任部门。
5. 不得为了让Mermaid图连贯而虚构节点。
6. 对缺失信息必须标记“待确认”。
7. Mermaid图必须能与流程节点表对应。
8. YAML对象必须保留来源和版本信息。
9. 生成文件必须符合仓库路径规范。
10. 每次生成必须输出文件清单和变更日志。
```

## 6.7 人工审核节点B：生成文件审核

### 审核位置

知识生成Agent完成Markdown、YAML、Mermaid后，必须暂停，进入人工审核。

### 审核内容

| 审核项 | 审核问题 | 处理方式 |
|---|---|---|
| Markdown | 是否准确承接抽取结果 | 不一致则退回修正 |
| YAML | 对象字段是否完整 | 缺字段则补充或标待确认 |
| Mermaid | 流程图是否能反映节点顺序 | 不准确则修正 |
| 风险控制 | 风险与控制是否保持原文 | 改写则退回 |
| 来源 | 是否保留source_file/source_location | 缺失则退回 |
| 路径 | 文件路径是否符合规范 | 不符合则调整 |
| 待确认 | 是否未把待确认内容当成正式内容 | 有则退回 |

### 审核结论

```text
通过：进入知识质检Agent。
有条件通过：修正指定问题后进入知识质检Agent。
退回：重新生成。
```

# 7. 知识质检Agent

## 7.1 Agent定位

知识质检Agent负责在入库语析前，对生成的知识文件进行一致性、完整性、可追溯性和可入库性检查。

它不负责修改正式内容，只负责发现问题、分类问题、提出处理建议。

## 7.2 输入

```text
1. 人工审核后的Markdown流程文档
2. YAML对象文件
3. Mermaid流程图文件
4. 基础问答片段
5. 文件清单
6. 抽取结果表
7. 人工审核记录
```

## 7.3 输出

输出《入库质检报告》。

必须包括：

```text
可入库清单
暂缓入库清单
问题清单
孤立对象清单
冲突清单
待人工确认事项
入库建议
```

## 7.4 工作流程

```text
步骤1：文件完整性检查
  - 检查应生成文件是否齐全。
  - 检查路径是否符合规范。
  - 检查文件命名是否符合规范。

步骤2：对象完整性检查
  - 检查每个对象是否有编号、名称、类型、来源、版本、状态。
  - 检查流程是否有治理域。
  - 检查风险是否有关联流程或节点。
  - 检查控制是否有关联风险。
  - 检查证据是否有关联控制。

步骤3：关系一致性检查
  - 检查Markdown、YAML、Mermaid之间是否一致。
  - 检查流程节点是否与Mermaid节点一致。
  - 检查风险控制关系是否一致。
  - 检查制度依据是否与对象文件一致。

步骤4：来源追溯检查
  - 检查每个对象是否有source_file。
  - 检查关键对象是否有source_location。
  - 检查是否存在无来源内容。

步骤5：Mermaid可渲染性检查
  - 检查语法是否明显错误。
  - 检查是否有未闭合节点。
  - 检查判断节点是否有分支条件。

步骤6：入库适配检查
  - 检查文档切片是否过大。
  - 检查元数据是否完整。
  - 检查是否适合语析知识库检索。

步骤7：生成质检结论
  - 可入库。
  - 暂缓入库。
  - 退回修正。
```

## 7.5 质检规则

必须检查以下规则：

```text
1. 每个流程必须属于一个治理域。
2. 每个流程必须至少包含一个流程节点。
3. 每个风险必须至少关联一个流程或流程节点。
4. 每个控制必须至少关联一个风险。
5. 每个控制原则上应有关联证据；如无证据，必须标记待确认。
6. 每个制度条款必须能关联流程、节点或控制中的至少一种。
7. Mermaid节点必须能在流程节点表中找到对应节点。
8. Markdown中的风险控制表必须与YAML对象一致。
9. 不得存在无来源对象。
10. 待确认事项不得作为正式结论入库。
```

## 7.6 边界规则

知识质检Agent必须遵守：

```text
1. 不得直接修改知识文件正文。
2. 不得自行补充缺失对象。
3. 不得自行合并重复对象。
4. 不得自行认定冲突以哪个资料为准。
5. 不得将低置信度内容判定为可入库。
6. 只能输出问题、风险等级、处理建议和入库建议。
7. 对需要人工判断的事项必须标记“人工确认”。
```

## 7.7 人工审核节点C：入库前确认

### 审核位置

知识质检Agent输出《入库质检报告》后，必须进行入库前人工确认。

### 审核内容

| 审核项 | 审核问题 | 入库处理 |
|---|---|---|
| 可入库清单 | 是否可以进入语析 | 确认后入库 |
| 暂缓入库清单 | 是否需要补充资料 | 补齐后再入库 |
| 问题清单 | 是否影响检索准确性 | 修正后再入库 |
| 冲突清单 | 是否需要指定权威来源 | 明确后再入库 |
| 低置信度内容 | 是否可作为待确认知识入库 | 原则上不入库正式库 |
| 待确认事项 | 是否已有明确结论 | 无结论不得入正式库 |

### 审核结论

```text
批准入库：将可入库文件导入语析。
部分入库：只导入确认无问题的文件。
暂缓入库：等待补充资料或修正。
退回重做：回到知识抽取或知识生成阶段。
```

# 8. 三个Agent之间的交付物衔接

```text
知识抽取Agent
  ↓ 输出：《知识抽取结果表》《冲突清单》《待确认事项》
人工审核节点A
  ↓ 输出：《审核后的知识抽取结果表》《审核结论》

  ↓ 输出：Markdown、YAML、Mermaid、问答片段、文件清单
人工审核节点B
  ↓ 输出：《审核后的知识文件》《生成文件审核结论》

  ↓ 输出：《入库质检报告》

  ↓ 输出：《入库确认结论》

```

## 8.2 阻断规则

以下情况必须阻断，不得进入下一环节：

```text
1. 抽取结果无来源文件。

3. 关键对象存在低置信度但未人工确认。
4. 流程节点与Mermaid图明显不一致。
5. 控制点没有关联风险。
6. 控制证据缺失但未标记待确认。
7. 待确认事项被写成正式结论。
8. 人工审核结论为退回。
```


```text
draft：草稿
reviewed：已人工审核
active：正式生效
pending：待确认
deprecated：废止
```
